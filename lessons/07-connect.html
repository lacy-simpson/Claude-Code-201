Give Claude reach

Connection Methods

Claude Code is capable on its own, and more capable once it can reach the systems you already use. Those connections come in two flavors. Knowing which is which helps you set them up and reason about what Claude can touch.

Lesson Objective

Tell the difference between connector-based and command-line connections, manage per-tool approvals, and apply a safety habit before letting Claude write to a system.

Two ways to connect

Connectors and Command-Line Tools

Connectors (MCP)

Ready-made connections that speak the Model Context Protocol, a shared standard for letting Claude use a system through well-defined tools. Some are hosted by Anthropic; some are built by teams. You add them from Customize → Connectors.

Command-line tools (CLI)

For systems without a connector, Claude can drive a command-line program you install, such as a cloud or database CLI. You install and authenticate it once, and Claude uses it to reach that system.

Adding a connector

What's Available, and How to Add It

Open Customize → Connectors to see what you can connect: common ones include Atlassian, GitHub, Glean, Google Drive, Microsoft 365 and Teams, Slack, Zoom, and Calendly. Connecting usually signs you in through single sign-on.

  • Add itFind the connector and add it. If something you want is not listed, you may be able to request it. Availability is set by your org admins.
  • Sign inMost connectors authenticate through single sign-on with your UiPath account.
  • Set approvals per toolFor read-only tools, you can set "always allow" so Claude does not stop to ask each time. Leave write or delete tools requiring approval.
A lock means your admin decided

Some tools show a lock icon and cannot be set to "always allow." That means an org admin has required approval for them, usually the actions that write or delete. You will approve those one at a time, by design.

Use the reach carefully

More Reach, More Care

Connections make Claude more useful, and they extend what it can change beyond your own laptop. That is worth a moment of caution. Read the actions you approve, especially anything that writes to a shared system, and confirm it is doing what you expect.

Test against a sandbox first

When a connection can write to a real system, try it against a sandbox or non-production environment first. A change that looks harmless can trigger downstream automations you did not expect. Prove it is safe where mistakes do not matter, then point it at the real thing.
