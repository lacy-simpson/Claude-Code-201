    Claude 201: Working with Claude Code

    Claude Code is the most capable way to put Claude to work on your own machine. By the end of this course you'll know your way around the desktop app, how to run and build skills, how to connect Claude to the systems you use, and how to keep yourself in control of everything it does.

    Self-paced reference All UiPath teams Hands-on with Claude Code

    If you have used Claude Chat, you already know how to converse with Claude. Claude Code takes the same assistant and gives it your files, your tools, and your permission to act. That extra power is why it deserves its own session: a little orientation goes a long way toward using it well and safely.

    What this course covers

    • Where Claude Code fits among Chat, Plugins, and Cowork, and a tour of the desktop app control by control
    • Choosing models and effort, reading your usage, and keeping sessions focused so Claude stays accurate
    • Permissions, prompting, memory, and other day-to-day habits that keep you in control
    • Connecting Claude to the tools you already use, and running tasks on a schedule
    • Running, building, and finding skills that do real work
    Who this is for

    Anyone at UiPath who wants to use Claude Code, on any team. You do not need to be an engineer. If a step assumes more technical comfort than you have, we flag it and give you a simpler path.

    Pick your surface

    Desktop App or Command Line

    Claude Code runs a few ways: as a desktop app with a full graphical interface, as a command-line tool (CLI) in your terminal, or inside an IDE like Visual Studio Code. They share the same engine.

    This course uses the desktop app. It is easier to set up, you can paste images straight into it, and the controls we walk through are visible on screen rather than memorized as commands.

    New to the terminal?

    That is completely fine. Stay on the desktop app for now. The CLI is excellent once you are comfortable in a terminal, and most of what you learn here carries straight over to it.

    Before you start

    Get Set Up

    Have these ready so you can follow along. If you are not set up yet, read through first and come back to the hands-on lessons afterward.

    • Install the Claude desktop app
      Download it from claude.ai/download for Mac or Windows, then open the Code tab inside the app.
    • Sign in with your UiPath account
      Use your UiPath enterprise account through single sign-on, not a personal account. This keeps your work in the company-managed environment.
    • Pick a folder to work in
      Create or choose a folder for your Claude Code work. You will point Claude at a folder when you start a session. We cover why this matters in the lessons ahead.

    New to Claude entirely? Start with Claude 101 for the basics of Chat, then come back here.

    01 Orientation 12 min
    Orient to Claude Code

    Claude Code & the Desktop App

    Claude shows up in a few different places, and each one fits a different kind of work. Before we open the app, it helps to see where Claude Code sits in that lineup. Then we tour the desktop window so the controls stop feeling mysterious.

    Lesson Objective

    Place Claude Code among the other Claude surfaces, describe what it can do that the others cannot, and find and name the main controls in the desktop app.

    Where it fits

    The Most Capable Surface

    The same Claude is available through several interfaces. They get progressively more capable as you move from a steered conversation toward a tool that can act on your machine.

    • Claude Chat — a conversation you steer turn by turn. It reasons, drafts, and summarizes, but it does not touch your files or systems.
    • Claude Plugins — Claude inside approved work tools, using the surrounding app context you allow: a thread, a spreadsheet, a doc, a deck.
    • Claude Cowork — a delegated-work mode that can read and write files and run skills. You hand off a bounded job and review the result.
    • Claude Code — the most capable surface, and the focus of this course. It reads your files, runs commands, and reaches other systems through tools, all with your permission.

    Claude 101 compares all four surfaces in depth, with a practice exercise for choosing between them. Start there if "which surface when" is still fuzzy.

    What is Claude Code?

    Claude Code is an agentic AI assistant that reads your files, runs commands, and edits work on your behalf, with your permission. It acts only where you point it, and when your permissions are set up well it waits for your approval before doing anything consequential.

    It's accessible

    Use it in the desktop app on the Code tab, in the terminal, or in your editor. The desktop app is the easiest place to start.

    It's interactive

    It is an agent, not just an advisor. It plans multi-step work and is shaped by the permissions and context you give it.

    It's extensible

    Skills, connectors, and command-line tools teach it a workflow once so you can reuse it across sessions.

    Know your workspace

    A Tour of the Desktop Window

    The Claude Code window has a handful of controls. Once you can name each one, the app stops feeling mysterious. This is a guided tour: we point at every control, say what it does, and note which lesson goes deeper.

    Start on the Code tab

    The desktop app has more than one mode. Make sure you are on the Code tab before you follow along. The Cowork tab looks similar but behaves differently.

    Where Claude works

    Local, and a Working Folder

    • Local vs remoteLeave this on Local. Claude Code can control another machine remotely, but for everyday work you want it acting on your own computer.
    • Working folderThis is the folder Claude can see and act in. Choose a different folder for each kind of work. Lesson 03 explains why one folder per purpose keeps Claude organized.
    • Additional foldersNeed Claude to also read files that live somewhere else, such as your Documents? Add them here so they are in reach without changing your main folder.
    Staying in control

    Permission Mode and Attachments

    • Permission-mode selectorSets how often Claude checks with you before acting. This is your main safety control. Lesson 03 covers each mode in detail.
    • AttachmentsAdd files to a message here: log files, a spreadsheet, an image. On the desktop app you can also paste an image straight into the input, which the CLI cannot do as easily.
    Capability and cost

    Model, Effort, and Usage

    • Model selectorChoose which model handles the task. Heavier models think more and cost more. Lesson 02 walks through Haiku, Sonnet, and Opus.
    • EffortA slider for how much reasoning Claude spends before answering. Medium to high is a sensible default.
    • Usage meterShows how much of your rolling and weekly limits you have used. Worth glancing at, especially on heavier models. Lesson 02 explains the limits.
    Working in parallel

    Sessions and Routines

    • New sessionStart a fresh session for each task with Ctrl/Cmd + N. Lesson 02 explains why one task per session matters.
    • Session listYour running sessions appear together. A session that needs your attention shows a small dot, so you can let work run in the background and check back. Open one to read its transcript or mark it complete.
    • RoutinesFound near the top of the window, routines let Claude run a task on a schedule and notify you. Lesson 04 sets one up.
    Key Takeaway

    Claude Code is the most capable Claude surface because it can act on your files and systems. Two controls matter most as you start: the working folder (what Claude can see) and the permission mode (what Claude can do without asking).

    02 Concept 12 min
    Capability, cost, and working memory

    Models, Usage & Context

    Two controls decide how much horsepower Claude brings to a task: the model and the effort level. They also decide how fast you move through your usage. And everything you do happens inside a context window, Claude's working memory for the session. A little intention with all three keeps Claude both capable and accurate.

    Lesson Objective

    Set the model and effort in Claude Code, read your usage so you know your headroom, and keep one task per session so the context window stays clean.

    Model and effort

    The Controls in Claude Code

    The model sets the level of capability Claude brings. Effort controls how much reasoning it spends before answering. Heavier choices can improve hard work, but they use more tokens and move you through your limits faster. The habit to build is simple: start with the lightest combination that can do the job, and move up only when the work needs it.

    Models change often

    Model names, defaults, and available settings change over time. Check the current Claude menu and UiPath guidance before choosing.

    Pick a model and an effort level. The readout shows what each choice changes in Claude Code. Start light, and move up only when the task needs it.

    Model
    Effort
    Thinking
    Move up on purpose

    If the first answer is too shallow, raise the model, the effort, or both. For most work, start with Sonnet at medium effort and reserve Opus for code and heavy building.

    Claude 101 has the full model-choice reference, Haiku vs. Sonnet vs. Opus, if you want the rationale behind each.

    Context size

    200K vs 1M Context

    Some models offer a one-million-token context, five times the default of two hundred thousand. Context is Claude's working memory for the task at hand. The larger window costs extra and uses your limits faster, so the default is the right choice unless you are genuinely working across a very large amount of material. Keeping each task in its own session, covered below, usually matters more than a bigger window.

    Know your headroom

    Reading Your Usage

    Claude Code shows your usage in the window. There are a few limits worth understanding so a heavy task does not surprise you.

    • 5-hour limitA rolling limit that starts counting from your first message in a session. How much you get can vary with overall demand, so it may feel tighter during busy hours.
    • Weekly limitA separate allowance that resets every seven days.
    • Extra usageUiPath provides additional headroom beyond those limits. If you hit a limit, you can keep working against that extra usage, and you can request more if you need it.
    Don't ration yourself into worse work

    The limits exist to be used. Pick the model and effort the task deserves rather than always defaulting to the lightest option to save tokens. If you run out, extra usage is there and more can be requested.

    What fills up

    The Context Window

    Everything in a session shares one context window: your messages, Claude's replies, the files it has read, and the system instructions it always carries. As a session grows, that window fills.

    When it gets close to full, Claude compacts the conversation. It summarizes what came before into a much shorter form so it can keep going. That summary keeps the gist, but detail is lost. If you were deep in a set of logs or a long draft, some specifics may quietly disappear.

    Compaction loses detail

    After a compaction, Claude may forget a specific value, name, or instruction from earlier. If an answer suddenly seems vaguer or contradicts something you established, compaction is a likely cause. Re-state the detail it lost and carry on.

    The habit that prevents it

    One Session per Task

    The most common way to fill the window too fast is running several unrelated tasks in one long session. Beyond using up the window, it invites confusion: details from one task bleed into another.

    Picture one session where you first draft a release announcement, then switch to analyzing a budget spreadsheet. The two share a window, so there is less room for each, and Claude can start mixing them: a number from the spreadsheet wanders into the announcement, or the tone of the email colors the analysis.

    • New session per taskStart a fresh session with Ctrl/Cmd + N for each distinct task. Each one gets the full window to itself.
    • Close when doneWhen a task is finished, let that session go rather than reusing it for the next, unrelated thing.
    • Reset if it driftsIf Claude starts referencing the wrong task, stop and start a clean session rather than fighting the confusion.
    Key Takeaway

    Start light on model and effort and move up on purpose, glance at your usage so nothing surprises you, and give each task its own session. One task, one session is the cheapest reliability habit you have.

    03 Habits 12 min
    Use Claude Code well

    Permissions, Memory & Ownership

    Claude Code is powerful because it can act. These habits keep that power pointed in the right direction: control what it is allowed to do, let it remember the stable facts about you, keep your work in focused folders, and keep ownership of the result.

    Lesson Objective

    Pick the right permission mode, recognize what each in-the-moment prompt is asking, set durable context with memory and folders, and keep responsibility for what Claude produces.

    Stay in charge

    Permissions: What Claude May Do

    You are always in charge of what Claude is doing. A permission mode sets how much it does before checking with you, and a permission prompt is the choice you make in the moment when it does stop to ask. Start with the modes, then see the prompts.

    Permission modes: how Claude runs

    Claude Code has session-level operating modes that control how much it does before asking. You can change the active mode from the toolbar.

    Ask
    Default
    What it means

    Claude asks for permission before every significant action: running commands, editing files, calling external services. Nothing happens without your say-so.

    Use when
    You are new, the work is sensitive, or you want to watch every step. This is your starting point for any unfamiliar session.
    Plan
    Review first
    What it means

    Claude writes a plan first and asks for the permissions it expects up front. You can revise the plan before it runs. It may still pause mid-run if it needs something new.

    Use when
    The work is multi-step and you want to approve the approach before anything happens, such as a refactor or a bulk update you cannot easily undo.
    Auto
    Less oversight
    What it means

    Claude auto-approves low-risk operations like reading files, searching, and safe commands, and still asks for higher-risk actions like writes or external calls.

    Use when
    A trusted, repetitive workflow you have already verified. Avoid it on a first run or when connected to production systems, since you will not see every intermediate step.
    Accept edits
    Edits auto-approved
    What it means

    Claude proceeds with reads and file edits on its own, and only stops to ask for actions that are not already allowed, such as commands or external calls.

    Use when
    An active session with many expected file changes and you want fewer interruptions. Review your changes afterward, since you will not see each edit in real time.
    Bypass permissions
    No guardrails
    What it means

    All permission checks are disabled. Claude acts without asking: no prompts, no diff previews, no safety gates.

    Use when
    Automated pipelines and sandboxed environments only. Never in an interactive session, where there is no undo and no review. When in doubt, use Ask.
    1 of 5 · Ask

    Permission prompts: what Claude is actually asking

    When Claude wants to take an action it pauses and asks. You will see one of these choices in the moment:

    Allow once
    Low risk
    What it means

    Permit this single action, this time only. Claude will ask again the next time it wants to do the same thing.

    Use when
    Your default for anything you have not done before. Safe when you want to stay informed as you build familiarity with a workflow.
    Always allow
    Use with care
    What it means

    Permit this kind of action for the rest of the session. Claude will not ask again; it just acts each time.

    Use when
    Only for low-risk, fully understood actions like a web search or reading a file you own. Never for writes to shared systems. When in doubt, Allow once is safer.
    Approve a plan
    Review the detail
    What it means

    When Claude proposes a plan or a change, it shows you what it intends to do before doing it. You can accept it, leave a comment to revise it, or deny it.

    Use when
    Anything that writes, updates, or deletes: files, records, messages, commits. If you could not undo it by hand in thirty seconds, see it before it happens.
    Deny
    Block it
    What it means

    Block the action entirely. Claude stops, explains what it was trying to do, and waits for new instructions from you.

    Use when
    Whenever something looks wrong or unexpected. Denying never breaks anything; it is always the safe choice. You can redirect Claude afterward.
    1 of 4 · Allow once
    Always allow the reads, approve the writes

    A common setup is to always allow read-only actions like web search, while still approving anything that writes or changes a system. You get speed where it is safe and a checkpoint where it counts.

    Durable context

    Personal Memory

    You can let Claude remember stable facts about you so you stop re-typing them. Turn it on under Settings → Capabilities → Memory. It is off by default.

    Once it is on, Claude builds up a picture of your role, your location, and the kind of work you do. After a while you can drop the standing context from your prompts because Claude already knows it. If you want a head start, you can import a description of yourself rather than waiting for it to build up.

    Keep it accurate, keep it sanitized

    Memory is only as good as what it holds. Add what is stable and useful, and avoid putting anything sensitive in it that you would not want reused across sessions.

    Stay organized

    One Folder, One Purpose

    Claude Code treats each folder as its own context, and it stores what it learns about your work per folder. Keep folders focused: a folder for a project, a folder for general conversations, a folder for a particular repository. Working from a focused folder means Claude builds useful memory about that work and does not see everything else on your machine.

    • A folder per kind of workStart sessions from the folder that matches the task, so the memory and files stay relevant.
    • Avoid starting from home or DesktopOpening a session at your home directory hands Claude visibility into far more than the task needs.
    Responsibility

    You Own the Outcome

    Claude assists you. It does not replace you, and it does not own the result. You do. Do not have Claude generate something and pass it on unread. Claude is also not deterministic: the same prompt can produce different answers, which is one more reason to read what it gives you.

    Key Takeaway

    Control what Claude may do, let it remember the stable context, keep your work in focused folders, and read the result before you use it. Those habits cover almost everything that goes wrong.

    04 Awareness 12 min
    Give Claude reach

    Connecting & Automating

    Claude Code is capable on its own, and more capable once it can reach the systems you already use. Connections extend what it can touch; routines let a task run on a schedule without you kicking it off. Both add reach, and both deserve the same care you bring to permissions.

    Lesson Objective

    Tell connector-based and command-line connections apart, manage per-tool approvals, set up a routine, and apply a safety habit before letting Claude write to a system or run unattended.

    Two ways to connect

    Connectors and Command-Line Tools

    Connectors (MCP)

    Ready-made connections that speak the Model Context Protocol, a shared standard for letting Claude use a system through well-defined tools. Some are hosted by Anthropic; some are built by teams. You add them from Customize → Connectors.

    Command-line tools (CLI)

    For systems without a connector, Claude can drive a command-line program you install, such as a cloud or database CLI. You install and authenticate it once, and Claude uses it to reach that system.

    Adding a connector

    What's Available, and How to Add It

    Open Customize → Connectors to see what you can connect: common ones include Atlassian, GitHub, Glean, Google Drive, Microsoft 365 and Teams, Slack, Zoom, and Calendly. Connecting usually signs you in through single sign-on.

    • Add itFind the connector and add it. If something you want is not listed, you may be able to request it. Availability is set by your org admins.
    • Sign inMost connectors authenticate through single sign-on with your UiPath account.
    • Set approvals per toolFor read-only tools, you can set "always allow" so Claude does not stop to ask each time. Leave write or delete tools requiring approval.
    A lock means your admin decided

    Some tools show a lock icon and cannot be set to "always allow." That means an org admin has required approval for them, usually the actions that write or delete. You will approve those one at a time, by design.

    Use the reach carefully

    More Reach, More Care

    Connections make Claude more useful, and they extend what it can change beyond your own laptop. That is worth a moment of caution. Read the actions you approve, especially anything that writes to a shared system, and confirm it is doing what you expect.

    Test against a sandbox first

    When a connection can write to a real system, try it against a sandbox or non-production environment first. A change that looks harmless can trigger downstream automations you did not expect. Prove it is safe where mistakes do not matter, then point it at the real thing.

    Work on a schedule

    Routines: A Task on a Timer

    A routine is a task you ask Claude to run on a schedule instead of kicking off by hand each time. You find routines near the top of the Claude Code window. A routine pairs a prompt with a schedule: every hour, every morning, or whatever cadence fits. When it runs, Claude does the task and can notify you, for example by sending a message in Slack or Teams, so the result comes to you instead of you going to look for it.

    A concrete example

    A Daily Folder Summary

    Say a shared folder collects reports through the day and you want a quick read each morning. Instead of opening it and skimming, you set a routine.

    • Write the task"Each weekday at 8am, summarize any new files in this folder into five bullets, and call out anything that looks urgent."
    • Set the scheduleChoose the cadence in the routine settings, such as every weekday morning.
    • Choose how you hear about itAsk Claude to send the summary to you on Slack or Teams so it lands where you already work.

    The same shape works for many jobs: a recurring digest, a periodic check on a status page, or a reminder built from whatever Claude can see.

    The same care applies

    A connection or a routine acts on your behalf, sometimes while you are not watching. Scope it tightly, prefer read-and-report tasks over ones that change systems, and everything from the permissions lesson still holds when the work runs on a timer.

    05 Hands-On 20 min
    Reusable playbooks for Claude

    Skills: Understand, Run, Build & Find

    Skills are how Claude Code stops repeating itself. Capture a task once and Claude follows the same playbook every time you ask. This lesson moves through the whole arc: what a skill is, how to run one, how to build your own, and where to find skills other teams have already made.

    Lesson Objective

    Explain what an Agent Skill is, install and run one, create a simple personal skill, and find a skill in the marketplace using any of the install methods.

    Understand

    What is an Agent Skill?

    A skill is a reusable playbook you give Claude for a specific task. Instead of explaining what you need from scratch every session, a skill captures it once: the steps, the rules, and the expected output. Claude picks it up automatically when your request matches.

    A skill is mostly just a folder with a text file. The most important file is SKILL.md. Its top section, called frontmatter, declares the skill's name and description. The rest is plain Markdown: the workflow, the rules, and examples.

    A skill on disk 
    meeting-notes-summarizer/
  SKILL.md          # the instructions Claude reads
  scripts/          # optional: code the skill can run for repeatable steps
    format.py
  references/       # optional: extra docs Claude loads only when needed
    Skills and scripts together

    Skills handle judgment, context, and variation: the things that change case by case. Scripts handle what is fast and fixed: retrieval, formatting, and predictable transformations. Together they are more reliable than either alone, because Claude does not have to reinvent the routine steps every time.

    Run

    Install One the Easy Way, Then Ask for It

    A skill is only useful once you run it, and running one is easy: install it, then ask for it by name. The simplest path needs no extra setup: download a skill as a ZIP, unzip it, and install it with one command. Start with a read-only skill so the first run cannot change anything.

    • Download and unzipDownload the skill's ZIP and extract it, for example into your Downloads folder.
    • Install itIn Claude Code, run the install command pointing at the unzipped folder.
    Install a downloaded skill 
    /plugin install ~/Downloads/meeting-notes-summarizer

    Once a skill is installed, you run it by asking. Claude recognizes the request and follows the skill's playbook, so a short request is enough.

    Run a skill 
    Run the meeting-notes-summarizer skill on the notes in this folder.

    A skill can take a moment, especially if it reads several files. A few things in the interface help you keep track without sitting and watching.

    • Expand the stepsClick into the messages to see what Claude is doing at each step, such as which files it is reading.
    • Let it run in the backgroundSome skills launch parallel helpers to work through many items at once. You can start another session and come back.
    • Watch for the attention dotA session that needs you shows a dot in the session list. Open it, see what it needs, and continue.
    • Review and closeWhen it finishes, read the output, then mark the session complete or move on.
    Build

    Create a Skill for Personal Use

    If you find yourself typing the same long instructions over and over, that is a skill waiting to happen. You can build one on your own machine, just for you, with no approval process and no sharing required. Good candidates are repetitive and a little fiddly:

    • A prompt you type constantly, with the same structure each time
    • A multi-step routine you would rather not re-describe
    • A formatting or summary job you want done the same way every time

    The easiest way is to ask Claude to build the skill for you. There is a skill-creator that scaffolds the folder and the SKILL.md, so you describe what you want and review the result.

    Ask Claude to build a skill 
    Create a personal skill called "weekly-update" that takes my rough notes
and turns them into a five-bullet status update for my manager.
Keep it local. Ask me anything you need before you write it.

    If you would rather write it by hand, you do not need much. A skill is a folder with a SKILL.md file. The frontmatter at the top names the skill and describes when to use it; the rest is plain Markdown.

    SKILL.md 
    ---
name: weekly-update
description: Turn rough notes into a five-bullet status update for a manager.
---

# Weekly update

When the user provides rough notes, produce a status update:

1. Read the notes and group them by theme.
2. Write at most five bullets, each one sentence.
3. Lead with what changed; end with any open risks.
4. Flag anything you inferred so the user can verify it.

    Building a skill is iterative. Your first version rarely handles every case, and that is fine. Run it on a real example, notice what is off (too long? missing the risks?), then tighten the SKILL.md or ask Claude to update it, and run it again.

    Find

    The UiPath Enterprise AI & Automation Marketplace

    You do not have to build every skill yourself. The UiPath Enterprise AI & Automation Marketplace is a centralized catalog of reusable AI assets: skills, automation agents, prompt libraries, and frameworks, vetted by your AI Automation Office. You browse by business domain, see what was published recently, or search for something specific.

    Each skill has a detail page so you can decide whether it fits before you install: an overview with use cases and tags, the current version and docs, and reviews from other people.

    A skill's page offers a few install methods. Pick the one that matches your setup.

    • Download ZIPNo repository access needed. Download the ZIP, unzip it, and install from the folder, exactly as in the Run section above.
    • Full CatalogDownload every skill in one bundle. Handy when you are getting started and want the whole set at once.
    • MarketplaceIf you have access to the internal repository, add the marketplace once, then install bundles by name. Updates flow through automatically.
    Marketplace: add once, then install bundles 
    /plugin marketplace add <marketplace>
/plugin install <bundle>@<marketplace>
    Key Takeaway

    A skill is a folder with a SKILL.md. Install one with a plain command, run it with a plain request, build your own when a task repeats, and check the marketplace before you build. Start with read-only skills and review what they do the first time, whether you wrote them or downloaded them.
