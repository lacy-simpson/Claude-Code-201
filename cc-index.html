What is Claude Code?

Claude Code is an agentic AI assistant that reads your files, runs commands, and edits code on your behalf, with your permission. It operates only where you point it, and when your permissions are configured correctly, waits for your approval before doing anything consequential.

It's accessible

You can access it through the Claude desktop app (</> Code tab), a terminal CLI, or directly at claude.ai/code.

It's interactive

Claude Code is an agent, not just an advisor. Give it a goal and it plans a multi-step approach, makes edits, runs commands, and iterates. What it can do is shaped by the permissions and configuration you set.

It's extensible

Skills, MCP servers, and plugins let you teach it your team's workflows once and reuse them across every session.

What is an Agent Skill?

A skill is a reusable playbook you give Claude for a specific task. Instead of explaining what you need from scratch every session, a skill captures it once: the steps, the rules, the expected output. Claude picks it up automatically when your request matches.

Skills shine for any task you repeat and where the output needs to be consistent. A few examples anyone might recognize:

Summarize a thread

Drop in an email chain or support ticket and get the key points, decisions, and next steps pulled out in a consistent format.

Draft a status update

Describe what you worked on and get a formatted update in the same structure every time, ready to paste into Slack or a report.

Review a document

Share any report, policy doc, or proposal and get a summary with the parts that need your attention flagged.

▸
Skills and scripts are better together Skills handle judgment, context, and variation — the things that change case by case. Scripts handle what's fast and deterministic: retrieval, formatting, fixed transformations. The combination is more powerful than either alone: offload the mechanical steps to a script, pass the result to a skill for reasoning and action. It's a pattern worth reaching for whenever a step is purely mechanical.

So what does a skill actually look like? Under the hood, it's just a folder with a few files. You don't need to memorize the structure right now, but it's worth a quick look so nothing feels mysterious later.

Anatomy of a Skill 
# A skill is just a directory with a markdown file
~/.claude/skills/weekly-report/
├── SKILL.md          # the instructions Claude reads
├── agents/
│   └── openai.yaml   # makes the skill discoverable in OpenAI clients
├── scripts/          # optional — local Python the skill can run
│   └── report.py
└── references/       # optional — extra docs the agent loads when needed

The most important file is SKILL.md. Its top section (called frontmatter) declares the skill's name and description. The rest is plain Markdown: workflow steps, rules, examples.

Create, Share, Improve

Most skills start as a personal fix for a repeated frustration. Then they get shared with the team. Then they get better. Here's how that arc typically looks.

1

1. Create a Skill

Anyone can create a personal skill and store it locally. It's just a folder with a SKILL.md file: no GitHub, no approval process, no pipeline. Write it once, drop it in your Claude Code config directory, and it's available in every session.

Good candidates for personal skills
  • Repetitive prompts you type constantly
  • Workflows specific to your accounts or territory
  • Personal formatting preferences or templates
  • Quick utilities that only you need
How to create one

The recommended approach is to use Claude's built-in skill-creator tool. Describe what you want the skill to do and it scaffolds everything for you.

You can also create one manually: add a SKILL.md file to ~/.claude/skills/my-skill/ and Claude Code picks it up automatically.

2. Share a Skill

Once a skill works well for you, ask whether it could help others too. If so, you can contribute it to a shared team repository. A shared repository lets the whole team benefit from each other's work, and lets everyone build on each other's skills over time. Many teams at UiPath keep a shared catalog for exactly this.

Good candidates for shared skills
  • Workflows more than one person on the team runs
  • Skills that solved a problem others have asked about
  • Prompts that benefit from team-wide consistency
  • Anything tested and stable enough to share
How to share one
  1. Start with a working personal skill.
  2. Ask a teammate to try it and give feedback.
  3. Follow your team repository's contributing guide: fork, branch, validate, and open a pull request.

3. Improve a Skill

Skills are rarely perfect after the first version. As you and your teammates use a skill, you'll spot places to tighten the instructions, sharpen the output, or cut steps that don't need Claude at all. Click each example to see what that looks like in practice.

You're using a shared meeting-summary skill. You and a teammate both run it on similar notes and compare results: yours comes back as a bulleted list, theirs as flowing prose. Neither of you changed anything.

The issue
The skill's SKILL.md never defines an output format. Claude infers a structure from context each run, so results vary across the team.
The improvement
You fork the repo, add an explicit output format requirement to the instructions, and open a PR. The team now gets consistent structure on every run.

You're using a shared log-analysis skill. It works, but every run feels slow and the token usage is higher than expected. You check the session transcript and see Claude spending several turns fetching and parsing the same structured data before the analysis even starts.

The issue
The log-fetch step is identical every run: same API call, same format. Claude isn't adding any judgment there — it's spending tokens on retrieval that a script could handle in milliseconds.
The improvement
You fork the repo, ask Claude to create a helper script that pre-fetches the logs, update the skill to use the pre-fetched data, and open a PR. Subsequent runs skip the retrieval overhead and go straight to analysis.

You just saw how Claude Chat, Claude Code, and Agent Skills fit together. Two questions to sit with before moving on:

  • What task in your week could become a skill? Most of us have at least one thing we do on repeat: summarizing a long thread, formatting an update, prepping for a recurring meeting. What's yours? You don't have to build it today, just name it.

Key Takeaways
  • Claude Code is the assistant. It runs in your terminal, the Claude Desktop App, or an IDE like VS Code, and acts on your behalf based on the permissions and settings you define.
  • A skill gives Claude a defined playbook for a specific task: structured, repeatable, and shareable if you choose to contribute it.
  • Skills can be personal or shared. Keep your own in your Claude config, and contribute the ones worth reusing to a shared team repository.

Lost context, compacted sessions, confused outputs: these common issues can be prevented. Let's take a look at recommended best practices that are the foundation of good Claude hygiene.

1 — Permissions: What You Can Allow Claude to Do, or Not

The three rules that follow only make sense once you understand the permission model. Take a moment to commit these three points to memory:

  • 1Claude Code has access to whatever folders you add to the workspace. It can read, write, and run commands there. Don't add folders you don't want it to touch.
  • 2MCP servers extend Claude's reach. When a tool (like Slack or GitHub) is connected via MCP, Claude can act through it. We'll cover connection methods in the next lesson.
  • 3You are responsible for what Claude does. Claude will prompt you for approval before taking actions, but only if you have it configured to do so.

Permission Prompts: what Claude is actually asking

When Claude wants to take an action (run a script, write a file, call an external service), it pauses and asks. You'll see one of these:

✓
Allow once
Low risk
What it means

Permit this specific action, this time only. Claude will ask again the next time it wants to do the same thing.

When to use thisYour default choice for anything you haven't done before. Safe to use when you want to stay informed as you build familiarity with a workflow.
⚠
Always allow
Use with caution
What it means

Permit this action type permanently for the current session. Claude will not ask again; it will just act each time.

When to use thisOnly for low-risk, fully understood actions, like reading a file you own. Never for writes to shared systems, record updates, emails, or anything shared. When in doubt, Allow once is always safer.
⚠ Easy to click by accidentIf Claude is writing to a shared system, running scripts, or calling APIs, don't select this until you've seen the action succeed correctly at least once and you fully understand what it's doing.
👁
Approve changes first
Recommended
What it means

Claude shows you exactly what it plans to do, the full diff or action detail, before doing anything. You confirm or cancel.

When to use thisAnything that writes, updates, or deletes: files, records, messages, repo commits. If you couldn't undo it manually in 30 seconds, you want to see it before it happens.
✕
Deny
Block it
What it means

Block the action entirely. Claude will stop, explain what it was trying to do, and wait for new instructions from you.

When to use thisWhenever something looks wrong or unexpected. Denying never breaks anything; it's always the safe choice. You can redirect Claude with a new instruction afterward.

Permission Modes: how Claude runs

Claude Code also has five session-level operating modes that control how much it does before asking. You can change the active mode from the toolbar or with /mode:

⊙
Ask permissions
Default
What it means

Claude prompts you for permission before every significant action — running commands, editing files, calling APIs. Nothing happens without your explicit approval.

When to use thisYour starting point for all sessions. Use Ask permissions whenever you're exploring an unfamiliar codebase, connected to production data, or learning what Claude would do.
📋
Plan mode
View only
What it means

Claude proposes a full action plan but cannot execute anything. No files change, no tools run. You review the plan and decide whether to proceed.

When to use thisActivate with /plan or press 3. Use before any high-stakes task — refactoring, bulk updates, anything you can't easily undo — to see the full scope before a single line changes.
⚡
Auto mode
Less oversight
What it means

Claude auto-approves low-risk operations (reading files, searching, running safe commands) and still asks for higher-risk actions like writes or external API calls.

When to use thisTrusted, repetitive workflows you've already verified. Never use on a first run or when connected to production systems — you won't see intermediate steps.
✏
Accept edits
Edits auto-approved
What it means

File edits are automatically accepted without a prompt. Claude still asks permission for commands, scripts, and calls to external services.

When to use thisActive development sessions with many expected file changes. Always review your git diff afterward — you won't see each edit in real time.
⊘
Bypass permissions
No guardrails
What it means

All permission checks are disabled. Claude acts without asking — no prompts, no diff previews, no safety gates. Enabled via the --dangerously-skip-permissions CLI flag.

When to use thisAutomated CI/CD pipelines and sandboxed environments only. Never in an interactive session — there is no undo and no review. If in doubt, use Ask permissions.

Golden Rules If Claude is about to touch something you couldn't manually undo in 30 seconds, select Approve changes first.
That means records, updates, emails, messages, anything in a shared repo.

2 — Prompt with intention

The output is only as good as the prompt. The acronym to remember is RTCFC.

RTCFC at a glance
  • Role — tell Claude who it is ("Act as a technical writer...")
  • Task — say exactly what you want done
  • Context — give it the relevant details: IDs, product, version
  • Format — specify the output shape ("email draft", "bullet list")
  • Constraints — set guardrails ("under 150 words", "no pricing details")
▸
Prompt with specifics, too Inside a session, vague prompts ("help me with this") force Claude to guess. Specific prompts ("Use the /weekly-report skill on this week's commits and format it as five bullets for Slack") activate the right skill and target the right thing. Specificity is part of hygiene.

3 — One folder, one purpose

Claude Code treats each folder as a separate context, like a separate repository. If you open it from your home directory or Documents, it can see everything there.

Do this instead
  • Create a projects/ folder and start each work session from the relevant project
  • Create a skills/ folder to use when you're working with skills
  • Never start a session from your home directory or Desktop

4 — One session per task

One of the most common mistakes is using one long-running session for multiple tasks. Claude accumulates context and eventually gets confused: its context window compacts and it starts making assumptions from the wrong task.

Do this instead
  • Open a new session for each task
  • When you finish a task, close the session
  • If Claude starts referencing the wrong task: stop, clear context, start fresh
!
Remember Your Responsibility Claude assists, but you own the outcome. Read every output, validate the content, make updates as needed before it reaches a customer, a ticket, or a shared doc.

Don't give Claude regulated or sensitive topics (HIPAA, SOC 2, pricing, SLAs, legal commitments).
▸
Before you move on: put the habits into practice

You just covered the habits that separate clean Claude sessions from messy ones. Two questions to sit with before the hands-on exercises:

  • Apply RTCFC to something real. Think of a task you do often, like a status update, a summary, or a first-draft message. How would you structure the prompt? What's the Role, Task, Context, Format, and Constraint? Take 30 seconds and sketch one out.
  • Which habit will be hardest to keep? One folder, one session per task, read every output before you share it: which of those is most likely to slip when things get busy? Why?

The goal isn't a perfect RTCFC prompt — it's starting to think in those terms before the hands-on exercises.

Key Takeaways
  • Claude has access to whatever you give it. Keep workspace folders small and specific — don't add anything you don't want it to touch.
  • Prompt with intention. Specific context produces accurate outputs. Vague prompts produce guesswork.
  • One folder, one purpose. Mixing projects or customers in a single workspace creates confusion and risk.
  • One session per case or task. Fresh sessions prevent context from one piece of work bleeding into another.
  • You own the output. Read every output, validate the output, and make adjustments before sharing it. Claude assists — the result is yours.

Claude can reach external services in three ways: Anthropic-hosted connectors available in Claude Desktop, team-built MCP servers configured by your org, and CLI-based authentication used directly by skills. Let's connect the dots before diving into the Connectors panel.

Claude Connection Types

Each method gives Claude a different level of access. The key difference is who configures the connection and what it can reach.

MCP under the hood
Anthropic-hosted
Ready to use, Anthropic-vetted
  • Available directly in Claude Desktop's Connectors panel
  • You authorize access once via OAuth — no config files needed
  • Anthropic defines and maintains the available tools
  • You are responsible for what Claude does with your account access
Team-built
Custom-configured, org-governed
  • Built and maintained by your org (e.g., your platform or tools team)
  • Admins define exactly which actions Claude is allowed to take
  • Tools are scoped to your team's specific workflows and systems
  • You are still responsible for every action you approve in the session

CLI-based authentication
CLI-based
Authenticated locally, skill-invoked
  • Some skills call a command-line tool (like the GitHub or Salesforce CLI) on your behalf
  • Access is scoped to your own account permissions
  • Runs entirely on your machine, with no Connectors panel involved

The tool runs with whatever access your account already has.

!
You are responsible for what Claude does — regardless of how it's connected. Neither method removes your accountability. Before running anything on a live system, make sure you understand what action Claude is about to take and who it affects.

Adding a Connector in Claude Desktop

Each connector uses OAuth: you authorize it once, and Claude can read from and write to that service on your behalf in any future conversation.

Follow along
  1. Open Claude Desktop and click Customize in the top left navigation.
  2. Click Connectors in the left navigation.
  3. Click the + icon at the top right of the Connectors panel to see available integrations.
  4. Select the connector you want, for example Slack or Microsoft 365.
  5. A browser window opens. Sign in and grant the requested permissions.
  6. Return to Claude Desktop. The connector is now listed as active.
  7. Open a connector that you configured and make note of the Tool permission settings for each interactive tool.

Repeat for each connector you want to enable. Start with the ones you use most: Atlassian, Slack, and Google Drive cover the majority of everyday workflows.

Safety Reminders

  • Always test before production. If a skill writes to a system or triggers a workflow, test it on a sandbox or non-production record first.
  • You own what Claude does. If a skill updates the wrong record, that is your error, not a Claude error.
  • When in doubt, ask. Check with your team before running anything on production you haven't tested.
  • Secrets stay out of sessions. Never paste API keys, tokens, passwords, or credentials directly into a Claude Code chat.
    Best practices and storage details
    • Use environment variables or .env files (in a folder Claude can't see)
    • If a secret is inside a file that gets committed to GitHub, use .gitignore to prevent it from ever being pushed
    • If you accidentally paste a secret: assume it's compromised, rotate it
    • For project credentials, use .claude/settings.local.json — Claude Code auto-gitignores this file; never hardcode credentials in prompts or in CLAUDE.md
    • Never put tokens in CLAUDE.md or any other committed file — they persist in version history
    • Claude Code does not auto-load .env files; inject env vars through settings.local.json if needed
    • Check your team's responsible-use policy for the full guidance
    iWhere does Claude Code store credentials, and which settings layer should I use?

    Your credentials are encrypted at rest. Claude Code stores API keys and OAuth tokens in ~/.claude.json, encrypted, with five rotating backups. They do not sit in plaintext on disk.

    Use the right settings layer for secrets:

    • .claude/settings.local.json — auto-gitignored by Claude Code. Use for personal or project secrets.
    • ~/.claude/settings.json — user-global secrets that apply across all your projects.
    • .claude/settings.json (checked in) — never put secrets here.

    Claude Code also supports hooks that can block access to specific files or paths. See Claude Code in Action for the deep dive.

    References on .gitignore: official git docs · W3Schools beginner guide

Claude Code is an AI assistant that runs in your terminal, the Claude Desktop App, or VS Code. It reads files, writes code, and runs commands in the folders you explicitly add to your workspace.

Agent Skills

A skill is a folder that gives Claude a defined playbook: what the task is, how to run it, and any scripts it needs. Claude loads the right skill when your request matches its trigger.

Skill vs. Python Script

Use a skill when…Use a script when…
Task needs natural-language judgmentTask is deterministic and repeatable
Output varies by caseOutput is always the same structure
You want Claude to reason about contextSpeed & reliability beat flexibility

Example: fetching logs → script. Drafting a summary → skill.

Connection Methods

Claude reaches external services three ways:

  • Anthropic-hosted connectors in Claude Desktop (OAuth, Anthropic-vetted)
  • Team-built MCP servers (org-governed, scoped to specific tools)
  • CLI-based auth used directly by skills

You are responsible for every action Claude takes, regardless of how it's connected. Keep secrets out of chat: use environment variables or settings.local.json, never plaintext.

Best Practices

Do

  • Prompt with intention: give Claude specific context, not vague requests
  • One folder per project
  • New session per task
  • Test changes before production
  • Use environment variables for credentials
  • Read every diff before you ship

Don't

  • Start sessions from ~/Documents or Desktop
  • Keep one session open all day for everything
  • Run untested skills on production data
  • Paste API keys or passwords into chat
  • Approve writes out of momentum

What does "MCP" mean?

MCP stands for Model Context Protocol. It's an open standard that lets AI tools (like Claude Code) talk to external systems (like Salesforce, Glean, Jira) through a small set of well-defined commands called tools.

You don't need to know the protocol details to use it. But you'll see "MCP server" referenced often. Think of it as a preconfigured bridge with defined parameters that lets Claude Code call into an external system on your behalf.
